# 第二天任务板

前提：服务器环境配置好，官方 Docker 能启动，三个 scene 能运行。

这份任务板只写目标和交付物，不限制具体实现方法。各组可以自由发挥，但晚上必须能说清楚自己跑到了哪一步。

Scene3 视觉与动作联调细节看：

[Scene3_Day2_视觉动作联调方案.md](Scene3_Day2_视觉动作联调方案.md)

## Day2 总目标

```text
Scene1：至少有 1 个包裹相关的可验证进展
Scene2：至少有 1 个零件相关的可验证进展
Scene3：至少有 1 个料盘相关的可验证进展
测试与提交：每个 scene 至少有一条记录
```

可验证进展包括：截图、视频、输出文件、日志、JSON、代码分支、运行记录等。

## 闫帅辰：视觉与 Scene3 坐标

负责范围：视觉读取、目标检测、深度/坐标、给动作模块的目标信息。

Day2 交付物：

```text
1. 一份 Scene3 视觉结果，能说明料盘在图像中的位置。
2. 如果 depth 跑通，给出一个 3D 坐标或深度结果。
3. 记录当前视觉方案的成功点和失败点。
```

不限制具体方法，可以使用 OpenCV、深度图、手工规则、其他轻量方法。

## 王森：Scene3 动作

负责范围：Scene3 料盘出库动作链路。

Day2 交付物：

```text
1. 一段 Scene3 动作尝试记录。
2. 说明手臂、夹爪、回撤等动作哪一步已验证，哪一步未验证。
3. 如果没有成功抓取，也要记录失败原因。
```

## 王仔俊：Scene1

负责范围：包裹称重与入箱，辅助视觉采样。

Day2 交付物：

```text
1. 一个包裹相关的识别、定位或动作尝试结果。
2. 一条 Scene1 测试记录。
3. 如果辅助采样，记录样本路径和对应 scene/seed。
```

## 史智涛 + Kevin：Scene2

负责范围：零件识别、分类、放入对应盒子。

Day2 交付物：

```text
1. 一个零件相关的识别、分类、定位或动作尝试结果。
2. 说明当前能区分哪些零件，不能区分哪些零件。
3. 一条 Scene2 测试记录。
```

## 窦欣悦：测试与提交

负责范围：测试记录、README、截图视频、最终提交材料。

Day2 交付物：

```text
1. 每个 scene 至少一条测试记录。
2. 每个人当天进度和卡点汇总。
3. 截图、视频、输出文件路径汇总。
4. 提交内容缺口列表。
```

## 晚上同步问题

每个人晚上回答这 5 个问题即可：

```text
今天负责什么？
今天实际跑到了哪一步？
有什么证据？
卡在哪里？
明天最小交付是什么？
```

## 合并标准

能合并到 main 的内容应满足：

- 路径清楚。
- 不依赖个人电脑绝对路径。
- 不提交大文件、rosbag、视频、build、devel、log。
- 至少有人在服务器上验证过或说明了未验证原因。
