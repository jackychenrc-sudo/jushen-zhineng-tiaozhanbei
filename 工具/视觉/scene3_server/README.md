# Scene3 上层料盘视觉、稳定判断与任务规划

这个目录保存闫帅辰负责的 Scene3「SMT 料盘出库」视觉、数据采集、机器人靠近和动作规划代码。代码仍处于联调阶段，不是完整比赛程序。

## 当前核心文件

### `scene3_upper_tray_perception.py`

**大概是什么：**

头部 RGB-D 相机的上层料盘感知程序。它综合多尺度外观匹配、深度竖直轮廓、货架平面和同层排列关系，从随机摆放的上层料盘中筛选目标，并输出像素框、深度以及 `base_link` 三维坐标。

**上传的意义：**

它把前期“只能在一张图片上粗略框选”推进到了实时 RGB-D 定位，可以给后续靠近和机械臂动作提供目标位置。V5 版本还区分模板识别框与 RGB-D 完整物体框，用于减少机器人靠近后把货架结构误认为料盘的问题。

**目前做到的程度：**

- 已在多个 seed、多个前后距离下检测上层 3 个料盘。
- 已输出相机坐标和 `base_link` 原始坐标。
- 已保存最终框图、全部候选图和 JSON 数据。
- 已加入近距离误框抑制，但当前近距离结果仍需继续人工核验和跨 seed 测试。
- 目前只处理上层料盘，还不能保证所有随机场景完全正确。

### `scene3_upper_approach.py`

**大概是什么：**

读取上层视觉 JSON 的短距离靠近程序。它先检查检测数量、综合分数、深度形状、RGB-D 几何来源和完整物体框；全部通过后，才能规划或执行一次受限的向前短脉冲。当前版本默认只接受已经通过多帧一致性检查的 JSON。

**上传的意义：**

它把“看见料盘”接到了机器人移动，形成“检测、短距离前进、停止、重新检测”的初步视觉闭环，同时避免一次走得过远。

**目前做到的程度：**

- 已完成 3 次真实短脉冲测试，视觉测得位移约为 `0.056 m`、`0.057 m` 和 `0.084 m`。
- 每次运动后都能重新检测并更新料盘距离。
- 当前只控制向前移动，不控制横移、机械臂和夹爪。
- 只有显式执行参数、确认口令和多帧稳定结果同时存在时才会运动；普通运行只是规划检查。
- 单帧结果只能通过调试参数做干跑检查，禁止用单帧模式执行机器人移动。
- 还没有完成料盘抓取、抽出、搬运和放入出库区。

### `scene3_multiframe_stability.py`

**大概是什么：**

连续读取至少 3 份 V5 感知结果，把不同帧中的同一个料盘按像素位置和三维位置配对，再检查目标数量、RGB-D 几何来源、得分、像素抖动和 `base_link` 坐标抖动。只有全部料盘都稳定时，才输出 `stable_upper_trays.json`。

**上传的意义：**

它把“某一帧刚好识别对了”提升成“连续多帧都识别到同一批稳定目标”，作为机器人继续靠近和以后伸臂前的安全门槛，减少单帧误检直接触发动作的风险。

**目前做到的程度：**

- 已支持直接读取已有 JSON，也支持在服务器上连续调用 V5 感知程序采集多帧。
- 默认要求至少 3 帧、3 个目标、像素最大波动不超过 12 像素、三维最大波动不超过 `0.02 m`。
- 已通过目标顺序打乱、位置小幅抖动、目标突然跳变、数量错误和坐标系变化等离线测试。
- 还没有在服务器上完成连续采集模式的实机联调，默认阈值后续仍需用更多 seed 验证。

### `scene3_upper_task_planner.py`

**大概是什么：**

读取 `stable_upper_trays.json` 的上层任务规划器。它判断机器人距离货架是过远、合适还是过近，并在距离合适且标定数据齐全时生成预抓取、张爪、抓取、闭爪、抽出、转移、释放和复核的干跑状态序列。

**上传的意义：**

它把视觉结果整理成动作组可以逐步验证的流程，同时借鉴旧行为树的 `EventArmMoveKeyPoint` 和 `control_leju_claw` 接口名称，但不照搬旧比赛的固定坐标和料盘位置。

**目前做到的程度：**

- 只生成 JSON 计划，`execution_enabled` 固定为 `false`，不会发送 ROS、机械臂或夹爪指令。
- 距离过远时只建议一次不超过 `0.08 m` 的短脉冲，并要求移动后重新做多帧识别。
- 距离过近、视觉不稳定或标定缺失时会明确阻止抓取计划继续。
- 机械臂偏移、姿态、出库区位置和夹爪参数仍为空，必须在当前比赛仿真中逐项标定后才能接动作执行器。
- 旧行为树接口尚未在当前比赛容器中确认可直接调用，目前仅作为适配参考。

### `scene3_upper_workflow_simulator.py`

**大概是什么：**

不依赖 ROS 和比赛服务器的上层任务闭环模拟器。它生成带有轻微视觉抖动的连续 RGB-D 检测结果，并复用仓库中的多帧稳定判断和任务规划器，模拟“识别、短步靠近、重新识别、停止或进入动作干跑”的流程。

**上传的意义：**

服务器被占用时，仍可检查状态转换和安全阻断是否正确，防止后续改视觉代码时出现漏检后继续前进、距离过近仍前进或未标定就进入抓取等逻辑回退。

**目前做到的程度：**

- 已覆盖正常靠近、完整动作干跑、视觉跳变、漏检和距离过近 5 类场景。
- 正常模式会在靠近后停在“等待真实标定”；只有明确的合成标定场景才会进入完整动作干跑。
- 视觉跳变、漏检和距离过近都会在发送动作前停止。
- 模拟器不会导入 ROS，也不会发送机器人、机械臂或夹爪命令。
- 该结果只能验证软件逻辑，不能验证动力学、逆运动学、碰撞、接触抓取和比赛得分；这些必须回服务器测试。

### `scene3_upper_calibration.example.json`

这是上层抓取标定配置的空模板。文件故意不填旧比赛坐标，防止把不同机器人、不同货架和不同场景的参数直接套到本比赛中。

## 输出文件怎么看

- `upper_trays.json`：最终入选料盘的框、得分、深度和三维坐标。
- `upper_trays.jpg`：最终入选结果，用于确认绿色框是否真的落在料盘上。
- `upper_candidates.jpg`：全部候选及其分数，用于分析漏检和误检。
- `stable_upper_trays.json`：连续多帧通过稳定性检查后的目标，靠近和动作规划应优先读取它。
- `scene3_upper_task_plan.json`：当前距离判断、目标选择、阻断原因和干跑动作顺序。
- `scene3_upper_workflow_simulation.json`：本地闭环模拟的每轮视觉、决策、抽象位移和终止状态。

视觉结果即使显示 3 个目标，也必须同时检查 RGB-D 几何来源和最终框图，不能只按数量决定机器人继续运动。

## 目录内其他辅助文件

- `scene3_dataset_collector.py`：按 seed 保存 RGB、深度、相机内参和 TF，供跨场景测试。
- `label_smt_trays.py`：手工选择料盘区域，主要用于早期模板和调试。
- `scene3_environment_probe.py`：检查相机话题、服务和 TF 是否可用。
- `scene3_live_perception.py`：早期实时 RGB-D 感知版本，为当前核心感知程序提供基础函数。
- `tests/`：多帧稳定门槛、任务规划器和闭环模拟器的 16 项离线回归测试，不依赖 ROS 或服务器。

原始数据集、运行日志和大量调试图片不上传 GitHub，只同步有复用价值的代码和阶段性说明。
